# Volume 1: Consistency Control — Document 2bis: SHACL Gate

## 1. Purpose and role

The SHACL Gate is the final validation stage of the consistency control
pipeline. It differs fundamentally from the three preceding gates in its
nature and objective.

The FATS, MATS, and OATS Gates are **primary consistency mechanisms** —
they detect and block inconsistencies before and during materialization,
operating on raw asserted triples without relying on inference. By the
time they have all passed, the data is expected to be clean.

The SHACL Gate is a **safety net**, not a primary mechanism:

- It operates on the full materialized closure MO (`<urn:kinship:full>`),
  after Materialization Step 2 has completed.
- It targets inconsistency categories that are structurally undetectable
  by the pre-inference SPARQL gates — either because they only manifest
  after inference, or because they result from errors in the
  materialization scripts themselves.
- A positive result (a violation) signals one of three root causes: a gap
  in the Step 1 materialization configuration (M incomplete), a bug in a
  materialization script, or a depth-limited pattern that escaped the
  SPARQL catalog.

A violation reaching the SHACL Gate should be treated as an architectural
signal, not a routine data correction. If the SHACL Gate fires repeatedly
for the same class of error, the correct response is to strengthen the
upstream gate that should have caught it, not to rely on the SHACL Gate
for primary detection.

---

## 2. Why SHACL rather than owl:Nothing

Triplestores do not treat OWL/RDFS constraints as integrity constraints
but as semantic rules intended for inference. Explicit violation detection
therefore requires an external mechanism.

`owl:Nothing` inference is not generated by GraphDB for constraint
violations. GraphDB's consistency rules detect violations but have no
triple consequences — they either silently ignore violations (default) or
reject transactions (`check-for-inconsistencies=true`). Neither produces
queryable triples.

SHACL is the appropriate mechanism because it:

- generates `sh:ValidationResult` triples that are stored and queryable
- runs against the materialized graph, including inferred triples
- produces structured diagnostics (`sh:focusNode`, `sh:resultMessage`,
  `sh:sourceShape`) that identify the involved individuals and the
  violated shape
- distinguishes severity levels (Violation vs Warning) natively

---

## 3. Named graphs

The SHACL Gate introduces one named graph and reads from two existing
ones:

| Graph | Role |
|---|---|
| `<urn:kinship:full>` (MO) | Full materialized closure — read by the validation engine |
| `<urn:kinship:shapes>` | SHACL shapes — loaded once, updated when shapes change |
| `<urn:kinship:validation>` | Validation report — written by the SHACL engine, cleared and repopulated on each cycle |

The shapes graph is loaded once at setup and does not change with each
cycle. The validation graph is cleared before each run to ensure the
report reflects the current state of MO.

---

## 4. What the SPARQL gates cannot detect

The SHACL Gate targets three residual categories that escape the
pre-inference SPARQL gates by construction.

### R1 — Interactions visible only in MO

Layer A reads M (closure of A alone) and O separately. It does not see
MO = closure(A ∪ O). Contradictions can emerge in MO when inferences from
the two sets combine in ways not visible to either gate individually.

This risk is bounded by the completeness of Materialization Step 1: if M
fully materializes `hasDescendant`/`hasAncestor` from A, Layer A Branch 1
(via `hasLineageRelative`) captures most of these interactions. Residual
R1 cases correspond to gaps in Step 1 configuration rather than gaps in
the gate logic.

### R2 — Errors introduced by materialization scripts

The gates validate A and O. If a materialization script produces an
incorrect triple in M or MO (a bug or a mis-specified rule), no SPARQL
gate sees it. Post-MO verification is the only available check.

### R3 — Depth-limited patterns in the SPARQL catalog

Q-MATS-PAR2 detects partnership+lineage conflicts at depth 2 (O(n³)).
Partners at greater genealogical distance are only detectable after
`hasDescendant`/`hasAncestor` are materialized, making the post-MO
check structurally simpler and more exhaustive.

## 5. SHACL shapes

Four shapes constitute the SHACL Gate. Each targets one of the residual
categories above. All run against `<urn:kinship:full>` (MO).

### ksh:PostCycleShape — generational cycle in materialized closure (R1, R2)

Detects self-ancestry via `hasDescendant` or `hasAncestor` in MO. A
positive result signals either a Layer A gap (incomplete Step 1
materialization) or a cycle introduced by a materialization script.

```sparql
## ksh:PostCycleShape
## Severity: Violation
SELECT $this WHERE {
    GRAPH <urn:kinship:full> {
        { $this kin:hasDescendant $this . } UNION
        { $this kin:hasAncestor   $this . }
    }
}
```

Root cause diagnosis:
- If the individual's blood parent chain forms a cycle in A → the MATS
  Gate Q-MATS-CIR2 should have caught it. Gap in the MATS Gate.
- If the cycle involves OATS assertions → Layer A or B did not catch the
  combination. Gap in the OATS Gate or Step 1 configuration.
- If neither → the materialization script incorrectly derived a cycle.
  Bug in the script.

### ksh:PostGenderShape — dual gender classification (R2)

Detects individuals simultaneously classified as both `MalePerson` and
`FemalePerson` in MO — a consequence of the GCI pattern
(`hasGender :Male → MalePerson`) applied to a person with two conflicting
gender assertions. The MATS Gate Q-MATS-CON3 should have blocked the
conflicting `hasGender` assertions in A. A positive result here signals
either a failure of Q-MATS-CON3 or a script that inferred class membership
incorrectly.

```sparql
## ksh:PostGenderShape
## Severity: Violation
SELECT $this WHERE {
    GRAPH <urn:kinship:full> {
        $this a kin:MalePerson .
        $this a kin:FemalePerson .
    }
}
```

Root cause diagnosis:
- Conflicting `hasGender` assertions in A → MATS Gate Q-MATS-CON3 gap.
- Correct `hasGender` assertions but incorrect GCI materialization → Bug
  in the GCI inference script.

### ksh:PostPartnerLineageShape — partnership in deep lineage (R3)

Detects partnership between individuals linked as ancestor/descendant in
MO. Q-MATS-PAR2 covers depth 2; this shape covers all depths using the
materialized `hasDescendant`/`hasAncestor` closure.

```sparql
## ksh:PostPartnerLineageShape
## Severity: Violation
SELECT $this ?partner WHERE {
    GRAPH <urn:kinship:full> {
        VALUES ?prop {
            kin:hasPartner kin:hasSpouse
            kin:hasCivilPartner kin:hasLifePartner
        }
        $this ?prop ?partner .
        { $this kin:hasDescendant ?partner . } UNION
        { $this kin:hasAncestor   ?partner . }
    }
}
```

Root cause diagnosis:
- Direct parent-child partnership → Q-MATS-PAR1 should have caught it.
  Gap in the MATS Gate.
- Grandparent/grandchild partnership → Q-MATS-PAR2 should have caught it
  at depth 2. Gap in Q-MATS-PAR2 or data at depth > 2.
- Greater depth → expected residual, correctly handled here.

### ksh:PostSiblingParentShape — sibling cascade from undetected cycle (R1, R2)

A generational cycle in A or in the MATS+OATS combination causes
`hasSibling` materialization to produce an individual as both sibling
and parent/child of the same person. This violates
`hasSibling ⊥ hasChild` / `hasSibling ⊥ hasParent`, which are declared in
`kinship-consistency.ttl`.

```sparql
## ksh:PostSiblingParentShape
## Severity: Violation
SELECT $this ?other WHERE {
    GRAPH <urn:kinship:full> {
        $this kin:hasSibling ?other .
        { $this kin:hasParent ?other . } UNION
        { $this kin:hasChild  ?other . }
    }
}
```

Root cause diagnosis:
- If the cycle exists in A → Q-MATS-CIR1 or Q-MATS-CIR2 should have
  caught it. Gap in the MATS Gate.
- If the cycle involves OATS assertions → Layer A or B gap.
- If `hasSibling` was materialized incorrectly → Bug in the materialization
  script.

## 6. Sentinel and coverage check

### Gate sentinel (Q-SHACL-EXISTS)

Before running the targeted shape queries, a single ASK establishes
whether any violation exists at all. If false, the targeted queries are
skipped entirely — the primary benefit on a consistent base is that the
O(1) sentinel short-circuits all subsequent O(n) checks.

```sparql
## Q-SHACL-EXISTS — sentinel
ASK {
    GRAPH <urn:kinship:validation> {
        ?r a sh:ValidationResult .
    }
}
```

### Ground truth count (Q-SHACL-COUNT)

If the sentinel returns true, the total count of distinct violating
individuals establishes ground truth for the coverage check.

```sparql
## Q-SHACL-COUNT — ground truth
SELECT (COUNT(DISTINCT ?node) AS ?violatingNodes) WHERE {
    GRAPH <urn:kinship:validation> {
        ?r a sh:ValidationResult ;
           sh:focusNode ?node ;
           sh:resultSeverity sh:Violation .
    }
}
```

### Targeted queries (Q-SHACL-*)

Each targeted query filters results by `sh:sourceShape` to identify the
specific category and the individuals involved.

```sparql
## Q-SHACL-CYCLE — generational cycle
SELECT ?node ?message WHERE {
    GRAPH <urn:kinship:validation> {
        ?r sh:focusNode ?node ;
           sh:resultMessage ?message ;
           sh:sourceShape ksh:PostCycleShape .
    }
}

## Q-SHACL-GENDER — dual gender classification
SELECT ?node ?message WHERE {
    GRAPH <urn:kinship:validation> {
        ?r sh:focusNode ?node ;
           sh:resultMessage ?message ;
           sh:sourceShape ksh:PostGenderShape .
    }
}

## Q-SHACL-PARTNERLINEAGE — partnership in deep lineage
SELECT ?node ?partner ?message WHERE {
    GRAPH <urn:kinship:validation> {
        ?r sh:focusNode ?node ;
           sh:resultMessage ?message ;
           sh:sourceShape ksh:PostPartnerLineageShape .
        OPTIONAL { ?r sh:value ?partner . }
    }
}

## Q-SHACL-SIBLINGPARENT — sibling cascade from cycle
SELECT ?node ?other ?message WHERE {
    GRAPH <urn:kinship:validation> {
        ?r sh:focusNode ?node ;
           sh:resultMessage ?message ;
           sh:sourceShape ksh:PostSiblingParentShape .
        OPTIONAL { ?r sh:value ?other . }
    }
}
```

### Coverage check (Q-SHACL-COVERAGE)

After all targeted queries, the combined count is compared to the ground
truth. A gap signals an uncovered violation category — either a new
residual pattern not yet shaped, or a regression in the pipeline that
produces a violation type the shapes do not yet recognize.

```sparql
## Q-SHACL-COVERAGE — gap detection
SELECT (COUNT(DISTINCT ?node) AS ?coveredNodes) WHERE {
    GRAPH <urn:kinship:validation> {
        ?r sh:focusNode ?node ;
           sh:resultSeverity sh:Violation ;
           sh:sourceShape ?shape .
        VALUES ?shape {
            ksh:PostCycleShape
            ksh:PostGenderShape
            ksh:PostPartnerLineageShape
            ksh:PostSiblingParentShape
        }
    }
}
```

A gap between `?coveredNodes` and `?violatingNodes` must be investigated
before the data is considered consistent. It does not block downstream
use of MO but should trigger a review of the shape catalog and the
upstream gates.

## 7. Pipeline preconditions and execution order

```
Precondition : MO available (Materialization Step 2 complete)
               <urn:kinship:shapes> loaded

Step 1        Trigger SHACL validation against <urn:kinship:full>
              Clear <urn:kinship:validation>
              Store sh:ValidationResult triples in <urn:kinship:validation>

Step 2        Run Q-SHACL-EXISTS
              If false → pipeline complete, MO is consistent ✓
              If true  → proceed to Step 3

Step 3        Run Q-SHACL-COUNT to establish ground truth
              Run Q-SHACL-* targeted queries by shape
              Run Q-SHACL-COVERAGE
              If coveredNodes = violatingNodes → all violations categorized
              If coveredNodes < violatingNodes → uncovered category, review required
```

## 8. GraphDB integration

SHACL shapes are defined in `consistency-shapes.ttl` and loaded into
`<urn:kinship:shapes>`:

```sparql
LOAD <file:consistency-shapes.ttl> INTO GRAPH <urn:kinship:shapes>
```

Validation is triggered via the GraphDB REST API after Materialization
Step 2:

```
POST /rest/repositories/{repoId}/shacl/validate
Content-Type: application/json
{ "shapesGraphUri": "urn:kinship:shapes" }
```

The validation report is stored for querying. In production this
insertion is automated by the pipeline orchestrator; the validation
graph is cleared before each run:

```sparql
CLEAR GRAPH <urn:kinship:validation> ;
INSERT INTO GRAPH <urn:kinship:validation> { ... }
```

## 9. Shape catalog — scope and limitations

| Shape | Targets | Residual category |
|---|---|---|
| `ksh:PostCycleShape` | self-ancestry in `hasDescendant`/`hasAncestor` | R1, R2 |
| `ksh:PostGenderShape` | dual `MalePerson`/`FemalePerson` membership | R2 |
| `ksh:PostPartnerLineageShape` | partnership + lineage at all depths | R3 |
| `ksh:PostSiblingParentShape` | sibling cascade from undetected cycle | R1, R2 |

**Out of scope by design:**

- **RED patterns** — redundant assertions are pre-inference data quality
  checks (MATS Gate step 6). Post-inference, the triples they detect are
  expected and correct — they are generated by the inference engine via
  `subPropertyOf` and `inverseOf` entailment. Including RED shapes would
  produce false positives on every correctly inferred triple.

- **FATS patterns** — FATS properties are rejected at ingestion and should
  never reach MO. A FATS property appearing in MO would indicate a failure
  of the FATS Gate, which is a more fundamental issue than the SHACL Gate
  is designed to address.

- **TWI1 (twins without common parent)** — no OWL axiom supports this
  constraint post-inference; it remains exclusive to Q-MATS-TWI1 in the
  MATS Gate.

- **NMA shapes (original design)** — the NMA-C1 through NMA-C4 shapes
  defined during earlier design iterations are superseded by the OATS Gate
  Layers A and B, which cover the same conflicts pre-inference with greater
  precision and without relying on materialized closures. They are not
  included in this specification.
